# DesignVault 三入口自动执行工作流

> 这页描述的是目标工作流，用于指导 skills、subagent、dashboard 和 automation 的公共实现。它不是“所有细节都已自动化完毕”的事实声明。

## 设计目标

- 把个人开发体验做成接近小团队协作的体验
- 让人类主要负责提出目标、做设计决策、最终试玩
- 让 AI 负责设计收敛、计划拆解、执行、验收、回写、汇总
- 尽量减少重复提示词、重复开线程和手动维护工作状态

## 三个入口

### `/design`

用于新功能、重构、设计收敛。

默认链路：

1. 读最小必要 wiki 真相
2. grill 问答把关键决策问清楚
3. 在单线程里收敛成长稿
4. 基于 `longform + wiki truth` 更新相关 wiki
5. 生成完整 `implementation plan`
6. 到 plan 结束

### `/execute`

用于从一个已写好的 `implementation plan` 自动推进到实现完成。

默认链路：

1. 读取 plan
2. preflight
3. 按 phase 线性执行
4. acceptance
5. 自动回写 wiki
6. 自动更新 execution log
7. 自动生成 dashboard 摘要

### `/bug`

用于从观察到的现象出发快速排查和修复问题。

默认链路：

1. 先读最小必要 wiki
2. 再查实现
3. 修复并验证
4. 如无设计变化，轻量更新说明与记录
5. 如发现本质是设计问题，停止并回 `/design`

## 文档分层

- `Longform`
  - 设计推敲层
- `Wiki Truth`
  - 当前稳定真相层
- `Implementation Plan`
  - `/execute` 的唯一主输入
- `Execution Log`
  - `/execute` 的工作历史和 phase handoff
- `Dashboard`
  - 人类主视图，只展示摘要

## Phase 定义

phase 不是：

- 人类中途验收单元
- 多人协作 DAG 节点

phase 是：

- 顺序执行单元
- subagent 上下文边界
- 自动 stop / handoff / writeback 的控制点

## 停机条件

以下情况必须停回主线程：

- `plan / wiki` 语义不一致
- `scene / prefab / wiring` 不确定
- bug 根因其实是设计问题

停机时返回一个短 `decision packet`。

## Acceptance Phase

默认对照：

- `longform`
- `wiki truth`
- 最终代码与测试结果

默认机器验收依赖：

- compile
- console
- targeted tests

## Repair Loop

如果 acceptance 发现偏差：

1. 先生成 `decision packet`
2. 人类决定是否追加 `Repair Packet`
3. 修完后只重跑受影响的 acceptance 部分
4. 必要时进入下一轮 `Repair Loop`

## 人类主视图

Dashboard 只回答这些问题：

- 最近完成了什么
- 我该试玩什么
- 有哪些待决策点
- 最近回写了哪些 truth 页面
