# 完整使用手册

这页是公开版 DesignVault 的完整操作手册。

## 一句话原则

`Longform 负责设计推敲，Wiki 负责当前真相，Execution Plan 负责工程实施。`

## 三类文档的职责

### Longform

- 偏设计语言
- 服务单线程持续推敲
- 允许保留开放问题、取舍和推理
- 不直接作为实现入口

### Wiki

- 偏稳定定义
- 是实现相关任务的起点和终点
- 服务人和 AI 快速查真相、查边界、查设计意图
- 不承载大段临时执行细节

### Execution Plan

- 偏工程语言
- 是完整 implementation plan
- 负责 phase、验证方式、回写目标、停机条件
- 必须引用 wiki，而不是重写 wiki

## 标准工作流

### `/design`

1. 先读相关 wiki
2. 通过提问把设计关键点问清楚
3. 在需要时沉淀成长稿
4. 把稳定内容写回 wiki
5. 基于 `Longform + Wiki` 生成完整 `Execution Plan`
6. 到 plan 完成时停止，不直接写代码

### `/execute`

1. 读取一个 `Execution Plan`
2. 做 `preflight`
3. 线性执行 phase
4. 用 phase handoff 保持上下文稳定
5. 做 `Acceptance Phase`
6. 写回 wiki
7. 写 `Execution Log`
8. 输出给人类的简要总结

### `/bug`

1. 从现象开始
2. 先读最小必要 wiki
3. 查实现并修复
4. 做最窄验证
5. 轻量更新 wiki 与执行记录
6. 如果根因是设计问题，停下来并回 `/design`

## Execution Plan 应该写什么

至少说清这些事：

- 这次要完成什么
- 真相从哪里读
- 最佳实现顺序是什么
- 每个 phase 开始时最少该读哪些信息
- 哪些情况必须停
- 验收大致怎么看
- 哪些页面可能要回写

允许自然语言。目标是让 `/execute` 稳定推进，而不是把作者逼成填表机器。

## Phase 的作用

- phase 是 AI 的执行边界
- phase 默认线性推进，不做并行依赖图
- phase 之间必须传递 handoff 信息
- 只有 handoff 成本很高时，才单独建 phase 页面

## Acceptance 与 Repair Loop

- `/execute` 默认以 compile、console 和 targeted tests 作为机器验收
- `Acceptance Phase` 对照 `longform / wiki truth / 最终代码与测试结果`
- 如果发现偏差，先生成 `decision packet`
- 人类决定是否进入 `Repair Loop`
- 修完后只重跑受影响的 acceptance 部分

## UI / Figma 约束

- 代码负责行为、状态、交互逻辑
- 最终样式不要硬编码在代码里
- 最终样式不要依赖运行时动态生成
- 视觉与样式尽量落在 prefab、scene、配置资产或其他编辑器可见资源中

## 维护能力

维护链路建议覆盖：

- lint
- wiki 检索
- writeback 建议
- change review 草稿
- dashboard 汇总

## 推荐 skills

- `designvault-design`
- `designvault-execute`
- `designvault-bug`
- `designvault-wiki-maintain`
- `designvault-ui-handoff`
